MainTitle

HTML es el lenguaje estandar para crear sitios y aplicaciones web, utilizado en conjunto CSS para el diseño y JavaScript para la interactividad. Los navegadores estan diseñados para interpretar el codigo de estas tecnologias.

Los elementos HTML son los bloques fundamentales que consitituyen toda pagina web, mientras que los archivos multimedia, reglas de CSS y Scripts de JavaScript se pueden agregar como bloques alternativos. La documentacion de todas las tecnologias relacionadas al lenguaje son recomendadas por el World Wide Web Consortium (W3C).

La especificacion HTML 2.0 fue publicada en 1995, al año siguiente fue publicada HTML 3.0 sin ser adoptada exitosamente. La especificacion HTML 4.0 fue publicada en 1997 y pasarian 17 años hasta que el estandar HTML 5.0 fue finalmente publicado en 2014.

  Historia

HTML fue un prototipo inventado en 1989 por Tim Berners Lee mientras trabajaba en el CERN de Geneva, Suiza. Su protocolo de transferencia HTTP permitia vincular dos documentos de forma remota y sin la necesidad de descargar cada documento por separado. El prototipo es instalado en una computadora NeXT (1990), la cual se convertiria en el primer servidor web.

La invencion del Domain Name System (DNS) permitio asignar un unico nombre a cada direccion IP, facilitando en gran medida la adopcion de HTML, el cual ahora podia distinguir facilmente entre los sistemas conectados a la red. HTML se baso en un inicio en los reglamentos internacionales relacionados con la transmision de documentos de texto, en especial en el SGML(Standard Generalized Mark-up Language). Algunos aspectos como los hypervinculos y los dominios fueron ideas de Tim.

1992: La NCSA (National Center for Supercomputer Applications) de Illinois, interesada por el protocolo de Tim, decide crear su propia aplicacion usando este lenguaje. Su navegador (Mosaic) es creado con la ayuda de Eric Bina y Marc Andreessen. Mientras tanto Tim abre una serie de discusiones publicas en Internet con el fin de crear la primer documentacion oficial de HTML.

1993: El navegador Lynx basado en texto es desarrollado por Lou Montulli, quien años despues seria contratado por Netscape. Lou pensaba que su navegador seria util sobretodo en terminales DOS que no permitian el uso de archivos multimedia. En este año Dave Raggett de HP viendo el exito de Lynx comienza su propio navegador titulado Arena.

1993: Marc Andreessen se sale de la NCSA y junto con Jim Clark fundan su empresa y su propio navegador titulado Netscape. La popularidad masiva del navegador fue en gran medida debido a su diseño que le permitia a personas con una coneccion a internet de baja velocidad acceder a internet.

1994: Debido a la creciente popularidad de la tecnologia las empresas lideres en el tema y las personas más interesadas en la estandarizacion del lenguaje se reunen para crear el W3C en las instalaciones del MIT. A la reunion asistieron representantes de HP, Microsoft, Digital Equipment, IBM, Netscape y Sun Microsystems.

En ese mismo año Microsoft lanza al mercado su navegador Internet Explorer, quien en los años siguientes se volveria el principal competidor de Netscape. En esta guerra de navegadores el W3C con Dave Raggett a la cabeza lucharia arduamente para que se mantuvieran ciertos estandares generales en torno a la tecnologia, culminando en 1997 con HTML 4.0 el cual establecia la relacion fundamental entre HTML, CSS y JS que sigue siendo utilizada hasta nuestros dias, perfeccionada hasta 2014 con HTML 5.0.

Despues de la creacion de HTML 5.0 y con el creciente enfoque del W3C por desarollar XML, Opera, Mozilla y Apple se unen para formar WHATWG el cual busca continuar en paralelo con la modificacion continua del HTML enfocado a nuevas tecnologias.

Los documentos HTML representan un DOM o arbol de objetos (nodos) en memoria. Los objetos estan delimitados por tokens y en donde cada objeto tiene una serie de atributos que dictan su comportamiento y contenido. El contenido de cada objeto se delimita utilizando etiquetas (tags), así mismo existen objetos cuyo contenido esta implicito y por tanto solo necesitan de un unico tag.

<!DOCTYPE html>
<html lang="en">
<head>
<meta charset="utf-8"/>
<title> Pagina Hola </title>
<link rel="stylesheet" href="styles.css />
</head>
<body>
<h1>Titulo</h1> 
<p><a href="https://www.w3.org/">Hello </a> World!</p>
<img src="http://www.w3schools.com/html/pic_mountain.jpg" alt="W3C" width="104" height="142" />
</body>
</html>

Los atributos se introducen en el tag inicial consistiendo de un nombre y un valor. Cuando solo se introduce el nombre se asume que el valor es un string vacio. El DOM, sus objetos y sus atributos pueden ser manipulados utilizando JavaScript.

var PrimerLink = document.links[0]; //Asigna el primer link del documento
PrimerLink.href = "sample.html"; //Cambia el destino del link
PrimerLink.protocol= "https"; //Cambia el protocolo
PrimerLink.setAttribute("href","http://ejemplo.com"); //Cambia el atributo completo directamente

Doctype

La Definicion del Tipo de Documento (DTD) o Doctype consiste en una declaracion que especifica los elementos y atributos permitidos de acuerdo a una especificacion dada, modificando el metodo de renderizacion (CSS) utilizado para visualizar correctamente el sitio.

Los navegadores actuales utilizan Parsers especializados en HTML por lo que las reglas de renderizado vienen incluidas y no es necesario especificarlas con documentos DTD. Cuando un navegador detecta el Doctype de HTML5, automaticamente utiliza el metodo de renderizado Standard mientras que versiones anteriores a esta declaracion activan el viejo metodo de renderizado Quirks.

//Sintaxis - URI in PUBLIC obligatoria cuando es XHTML.
<!DOCTYPE Root-Element PUBLIC "FPI" "URI"> 
<!DOCTYPE Root-Element SYSTEM "URI">

//HTML 5+
<!DOCTYPE html>
//HTML 5+ Alternativa
<!DOCTYPE html SYSTEM "about:legacy-compat">
//HTML 4.01 Strict
<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.01//EN" "http://www.w3.org/TR/html4/strict.dtd">
//HTML 4.01 Transitional
<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.01 Transitional//EN" "http://www.w3.org/TR/html4/loose.dtd">
//XHTML 1.0 Strict
<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd">
//XHTML 1.0 Transitional
<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">
//HTML 3.2
<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 3.2 Final//EN">
//HTML 2.0
<!DOCTYPE html PUBLIC "-//IETF//DTD HTML 2.0//EN">
//XML Declaration (Standalone="No" -> External DTD)
<?xml version="1.0" encoding="UTF-8" standalone="no" ?>

Las versiones Strict tienen reglas de renderizado que evitan que se utilice HTML para modificar el diseño del sitio, por otro lado las versiones Transitional contienen reglas de renderizado permisivas. La forma alternativa (No recomendada) de escribir el Doctype HTML5, fue diseñada para software y Parsers que necesiten obligatoriamente de un Indicador para funcionar correctamente.

Un Indicador es un Keyword ideado en SGML para que el Parser del documento localice la entidad que dicta las reglas de renderizado. En SGML, despues de los identificadores, se pueden agregar declaraciones opcionales que modifiquen los valores iniciales del Doctype, esta sintaxis no se incluyo posteriormente en HTML.

SYSTEM: Indicador del Sistema usado cuando asumimos que el sistema que recibe el documento no tiene conocimiento de la ubicacion de las entidades necesarias para renderizarlo correctamente. El Identificador del Systema URI, usado por el Indicador del Sistema, utiliza entonces direcciones absolutas.

PUBLIC: El indicador publico utiliza un FPI y opcionalmente un URI para localizar a la entidad. El FPI consiste de la institucion que emite la entidad, el tipo de entidad, el nombre de la entidad y el lenguage en el que esta escrito el codigo source de la entidad. Podemos relacionar un FPI a un URI por medio de un archivo "Catalogo" actualizado que tradusca el FPI en su URI equivalente.

Cuando queremos que un documento sea renderizado como verdadero XML, tenemos que usar la Declaracion XML antes del Doctype. En este caso dado que el Identificador no puede tener prefijos, la direccion debe ser relativa al documento XML, es decir que la entidad debe de estar en los archivos locales del sistema que recibe el documento.

URL Code

URL o Uniform Resource Locator es un codigo estandar utilizado para encontrar un recurso en un una red interna o remota. Es usado comunmente a travez del protocolo de internet (HTTP) para vincular sistemas (dominios) y los documentos que almacenan sin tener que utilizar su Direccion de Protocolo de Internet (IP).

Un URL se compone de un Esquema (Protocolo de Transferencia), Prefijo (Prefijo del Dominio), Dominio (Nombre Registrado), Puerto (El puerto estandar para internet es el 80), Path (Direccion del recurso en el dominio), Archivo (Nombre del archivo final). Por default si se omite el Archivo se intenta abrir la carpeta superior en el Path. Si se omite el Archivo y el Path entonces se busca el archivo en el dominio (index).

Un URL es una direccion valida siempre y cuando se utilice ASCII sin espacios. En el caso de que el dominio, el path o el archivo utilicen caracteres fuera de la especificacion ASCII se recurre a una encoding propio de la especificacion URL.

El enconding de URL remplaza todo caracter que no sea ASCII por un equivalente hexagesimal despues de el caracter de escape (%). Los espacios son remplazados normalmente con un signo (+) o con (%20).

Un recurso local (src) puede estar localizado en el mismo folder (imagen.jpg), en una carpeta anexa (images/imagen.jpg), un el folder de imagenes relativo a la raiz del sitio (/images/imagen.jpg) o un nivel arriba con respecto al folder actual (.../imagen.jpg). En general se recomienda usar URLs relativos al sitio.

Existen varios esquemas de trasnferencia:

    scheme:[//[user:password@]host[:port]][/]path[?query][#fragment]

<!--
Scheme: Protocolo como http, https, ftp, mailto, file, data y irc.
Las dos lineas diagonales son utilizados por algunos esquemas.
Autentificacion opcional para accesar al servidor.
Servidor Host compuesto del nombre registrado o su direccion IP.
Puerto opcional por el cual transmitir la informacion (80 default).
Path y Archivo que se esta buscando en orden jerarquico.
Query Delimiter: Busca y encuentra recursos no jerarquicos.
Fragmento opcional que encuentra un identificador interno en el documento.
-->

    Para solucionar problemas de internacionalizacion se utilizan IRI (Internationalized Resource Identifier), que es un tipo de URL que incluye caracteres unicode. Todos los navegadores soportan el uso de IRIs. El nombre registrado del dominio en IRI se conoce como IDN (Internationalized Domain Name). La mayoria de los servidores web convierten automaticamente cualquier vinculo con caracteres no ASCII en codigo registrable por los DNSs, normalmente escapando los caracteres con (xn--).

    Si el URL no es convertido automaticamente por el servidor, entonces el navegador utiliza el encoding nativo de URLs (%) para encontrar la pagina original al decodificar la direccion escapada.

    Mime type

    Un Media Type o tipo de medios, tambien denominado MIME Type o Content Type, es un identificador de dos partes para determinar el formato o caracteristicas de un archivo que se transmite por internet.

    Parte de la especificacion MIME, originalmente diseñada para controlar los tipos de archivos que se compartian por el protocolo de email, pronto seria adoptado por el protocolo HTTP.

    Se compone de un Type (tipo), Subtype (subtipo) y parametros opcionales; tambien se puede escribir como un Type, Tree, Sybtype, +Suffix y Parametros. Por ejemplo, antes de la llegada del tag Charset se acostumbraba definir el encoding con el parametro de un MIME Type (text/html; charset=UFT-8).

    Actualmente solo existen diez tipos primarios: Application, Audio, Example, Image, Message, Model, Multipart, Text y Video. Aunque todavia no es oficial se ha llegado a utilizar el tipo Chemical. Actualmente los Suffix registrados son: xml, json, ber, der, fastinfoset, wbxml, zip, cbor. Si no se conoce el tipo se suele utilizar de forma generica (text/plain), de igual forma en documentos binarios sin tipo especifico (application/octet-stream) es utilizado.

    Los navegadores solo reconocen los subtipos de imagenes: Gif, Jpeg, Png, Svg+xml, Webp, x-icon. Solo reconocen los subtipos de video: webm, ogg. Solo se reconocen de audio: wave, wav, x-wav, x-pn-wav, webm, ogg. En este sitio de IANA se encuentra la lista completa.

    Html

    Primer elemento del DOM (Root Element), le informa al navegador que el documento es especificamente HTML. Todo elemento debe derivar de este, se recomienda que los unicos dos elementos hijos sean las etiquetas (Head) y (Body).

    Atributos:

    1. Manifest: Especifica el URI de un archivo Manifest que indique los recuros que deben ser almacenados localmente utilizando el App Cache del sitio. El atributo solo es considerado en la etapa Load, por lo que cambios dinamicos posteriores no tienen efecto. JS puede accesar al cache utilizando el IDL windo.applicationCache, sin embargo esta depreciado y se recomienda usar en su lugar Service Workers.

    2. Version: Especifica la version del Doctype, este atributo es redundante y no se debe de utilizar.

    3. Xmlns: Si se esta escribiendo un documento XML que sera renderizado por XML parsers se necesita especificar el Namespace (http://www.w3.org/1999/xhtml). Si es un documento XHTML este atributo es opcional.

    Atributos Opcionales:

    • lang ó xml:lang: Especifica el lenguaje primario del elemento, usado generalmente en la etiqueta (html). Podemos usar las listas BCP47, ISO639-1 o IANA para obtener los tags de cada lenguaje. Si estamos escribiendo HTML solo usar (lang), si estamos escribiendo XML (application/xhtml+xml) solo usar (xml:lang). Si estamos escribiendo XHTML o HTMLPolygot usar ambos.

    Head

    Contiene la informacion general o Metadata del documento que no se renderiza en el navegador. La metadata puede incluir informacion general (meta), dependencias (links), scripts (script), estilos (style), entre otros. Se recomienda incluir siempre un unico titulo con la etiqueta (title) y nunca más de una etiqueta (base).

    Atributos:

    1. Profile: Especifica un URI para localizar uno o más perfiles metadata externos. El atributo esta depreciado y se recomienda usar en su lugar (rel="profile") dentro de una etiqueta (link).

    Meta

    Elemento que representa la informacion general o metadata del documento que no pueda ser representada usando las demas etiquetas metadata.

    Atributos:

    1. Name: Nombre del dato metadata a asignar, este puede ser Application-name, Author, Description, Generator, Keywords, entre otros.

      1. Application-name: Si un sitio es una aplicacion web dinamica se puede introducir el nombre de la aplicacion con este atributo.

      2. Author: Es un string con el nombre del autor de la pagina, si hay varios autores se puede repetir este elemento.

      3. Description: Descripcion del sitio utilizando alrededor de 100 palabras, esta descripcion es utilizada por los buscadores.

      4. Generator: String con el nombre del generador o software usado para crear el sitio, en el caso de que haya sido generado automaticamente.

      5. Keywords: Tokens separados por comas que representan las palabras más relevantes del sitio, utilizado con precaucion debido a su uso indiscriminado en el pasado.

      6. Viewport: En proceso de aceptacion, controla la escala del zoom para ajustar el ancho de los elementos en navegadores movil.

    2. Http-equiv: Directiva Pragma que modifica al sitio simulando un Http-Response-Header-File, este puede ser Content-type, Default-Style o Refresh.

      1. Content-type: Usado antes de HTML5 para determinar principalmente el encoding del sitio (Content="text/html; charset="UTF-8"). Ahora se recomienda utilziar el atributo Charset.

      2. Default-Style: Si existen varios documentos CSS, este atributo puede señalar el titulo (Title) del elemento (Link) con el documento de estilos preferido que se pretenda utilizar. Sin embargo se recomienda evitar situaciones en donde se utilicen varios documentos CSS.

      3. Refresh: Redirige a un sitio externo despues de un tiempo establecido en segundos. Si solo se especifica el tiempo entonces se recarga el mismo sitio. Este atributo esta depreciado y se recomeinda utilizar Headers HTTP Reales.

    3. Content: Asigna el valor al dato metadata o a la directiva.

    4. Chartset: Determina uno de los encoding usados en los caracteres del sitio cuando esté se transmita de forma serializada. No es tomado en cuenta en XHTML pero es valido. Localmente el atributo (Lang), el (Http-Content-Type) y el (BOM Header) tienen prioridad. Se recomienda que este dentro de los primeros 1024 Bytes del sitio.

    Title

    Contiene un string con el titulo del documento, mismo que se muestra en la barra superior de los navegadores. Es muy importante incluir esta etiqueta correctamente en el (Head) del documento ya que es requetido por los navegadores para renderizar el sitio. Solo debe de haber uno por documento.

    Se puede modificar su contenido con (document.title="new title") u obtener su contenido con (var a = title.textConent).

    Link

    Vincula el documento actual con un recurso externo, generalmente un archivo de estilos CSS.

    Los atributos (onload) y (onerror) pueden llamar a una funcion de JS cuando se termino de cargar el recurso y si se detecto un error de ejeucion o renderizado.

    Atributos:

    1. Crossorigin: Activa el modo CROS, generalmente para elementos canvas.

    2. Href: Especifica el URL del recurso externo.

    3. Hreflang: Especifica el lenguaje del recurso. Su contenido es meramente informativo, pocas tecnologias lo utilizan como una declaracion.

    4. Integrity: Contiene el Hash utilizado por el sistema de seguridad Subresource. SRI permite comprobar que el recurso externo no haya sido manipulado al comparar este hash de 64bit que habia sido generado con el archivo externo.

    5. Media: Especifica los atributos media para ejecutar el documento. Los atributos media nos permiten aplicar estilos especificos dependiendo del dispositivo y son la principal herramienta para un Responsive Design. Por defecto su valor es (all).

    6. Title: Especifica si el archivo es persistent, preferred o alternative. Si no tiene este atributo el archivo se aplica siempre (Persistent). Si varios tienen este atributo y el mismo (rel) entonces el navegador decide cual aplicar. Por ultimo para diferenciar de un archivo persistent (Default) y uno alternativo se utiliza el atributo (rel="alternate stylesheet").

    7. Type: Utiliza los MIME tipes para definir que tipo de documento es, por ejemplo HTML (text/html) o CSS (text/css).

    8. Rel: Especifica la relacion del recurso con el documento.

      1. Alternate: Generalmente rel tiene el valor de (Stylesheet) sin embargo podemos asignar stylesheets opcionales con el valor de (alternate stylesheet).

      2. Stylesheet: Su uso más comun es simplemente para establecer la relacion con documentos de diseño.

      3. Preload: Activa una tecnologia experimental que especifica recursos de alta prioridad en el rendering de la pagina, así el sitio puede priorizar dinamicamente el contenido que renderiza de forma asincrona. Se usa en conjunto con el atributo (as="style...").

      4. Prefetch: Cuando el navegador encuentra hiperlinks que tengan este atributo, intentara en la medida de lo posible cargar los sitios a los que se dirigen en el cache del navegador. Normalmente se usa en los links más usados para mejorar la experiencia del usuario. Tambien se puede usar para guardar en cache unicamente la resolucion DNS de dichos links, no su contenido.

      5. Prerendering: Se pude considerar como una version extrema de prefetch en donde se guarda en el cache la version completa del recurso o sitio al que se vincula. Aunque el hipervinculo se puede volver casi instantaneo, se debe usar con cuidado por su uso intensivo del bandwidth.

    Body

    Todo contenido que se pretenda mostrar en el navegador se debe incluir en esta etiqueta. Junto con (html) y (head) es una etiqueta fundamental para cualquier sitio web.

    Atributos:

    1. Onafterprint: Llama una funcion cuando el usuario intenta imprimir el sitio.

    2. Onbeforeprint: Llama una funcion cuando el usuario termino de imprimir el sitio.

    3. Onbeforeload: Llama una funcion cuando el usuario intenta cerrar el sitio, normalmente es un mensaje de confimacion.

    4. Onblur: Llama una funcion cuando el usuario cambia de pesataña, dejando al sitio en el fondo, sin embargo su uso principal en es formas para ejecutar funciones conforme el usuario pasa al siguiente campo

    5. Onfocus: Llama una funcion cuando el usuario cambia a la pestaña del sitio, sin embargo su uso principal en es formas para ejecutar funciones conforme el usuario llega a un campo.

    6. Onerror: Llama una funcion cuando el sitio no su puede renderizar correctamente.

    7. Onload: Llama una funcion cuando se termino de cargar el sitio.

    8. Onoffline: Llama una funcion si se detecta la perdida de coneccion a internet.

    9. Ononline: Llama una funcion si se detecta la reconeccion a internet.

    10. Onresize: Llama una funcion si se detecta un cambio en el tamaño de la ventana del sitio.

    H1-H6

    Representan los titulos de las secciones en un documento y su jerarquia o nivel. Los navegadores y buscadores los utilizan para estructurar el contenido, se recomienda cambiar de nivel unicamente cuando necesites jerarquizar el contenido, en el caso de que el contenido sea el mismo es recomendado modificar su apariencia con CSS en vez de utilizar un titulo de otro nivel.

    Se recomienda nunca saltarse un nivel, normalmente es posible ir bajando paulativamente de seccion en seccion. En general un sitio web utiliza un titulo (h1) general y despues divide el documento en secciones con los demas niveles.

    En HTML con la invencion de la etiqueta (Section), se recomienda que todo el contenido de cada seccion utilice un mismo titulo (e.j. h2) y en el caso de querer más subdivisiones se pueden utilizar titulos de menor nivel (e.j, h3); Nested Sections heredando el titulo de su padre (h2) o combinar ambas tecnicas utilizando titulos de menor nivel para cada nueva nested section.

    P

    Este elemento representa un parrafo de texto. Visualmente son bloques separados verticalmente por un espacio, en muchos casos existen porciones de texto en donde existen etiquetas más apropiadas para definir su estructura.

    A

    Este elemento fundamental genera un hypervinculo entre sitios y documentos en linea utilizando su URL. Un hypervinculo no esta limitado al protocolo HTTP y puede vincular documentos a travez de FTP, File y Mailto.

    Atributos:

    1. Title: Atributo generico utilizado para especificar un Tooltip de ayuda.

    2. Download: Instruye al navegador a descargar el documento sin importar su extension.

    3. Href: Atributo en donde se especifica el URL relativo o absoluto con el recurso externo o seccion interna (URL Fragment). Si no se usa este atributo se generara un Dummy Link que no funciona. Por razones historicas tanto (href="#top") como (href="#") vinculan a la parte superior de la ventana.

    4. Target: Inicialmente utilizado con Iframes, especifica el lugar en el que se abre el vinculo, ya sea en la misma ventana (Self), en una nueva pestaña (blank), en la ventana padre (parent), o en la ventana principal (top).

    5. Type: Especifica el tipo MIME del documento, sin embargo es un atributo meramente informativo.

    6. Rel: Especifica la relacion del recurso con el documento.

      1. Author:

        El link vincula a un documento que trata del autor del sitio.

      2. Help:

        El link vincula a un documento de ayuda.

      3. Licence:

        El link vincula a un documento legal con la licencia del sitio.

      4. Next:

        El link vincula al siguiente documento cuando se esta en un sitio por partes, diapositivas o paginas de libro.

      5. Prev:

        El link vincula al anterior documento cuando se esta en un sitio por partes, diapositivas o paginas de libro.

      6. Search:

        El link vincula a un sitio de busqueda interna.

      7. Pingback:

        Ayuda a sistemas automaticos a avisarle al sitio de destino acerca de la vinculacion. En WordPress es usado para informarle a un blog que otro blog lo ha vinculado.

    Img

    Elemento que representa una imagen.

    En el caso de un error, el evento (onerror) es llamado, algunos de estos casos incluyen un src mal escrito, la imagen esta corrupta, su metadata es incorrecta o el formato no esta soportado por el navegador.

    Una imagen pueden ser estatica (Png, gif, jpeg), en vectores (PDFs de una pagina, XML, SVG), animadas en bitmap (APNGs, GIFs) o animadas en vectores (XML con SVG en formato SMIL). Se prohibe el uso de SVG+Script, PDFs multipagina, MNG, HTML y texto.

    Dependiendo de el estado de la informacion y que tanto de la imagen ha logrado renderizar un navegador, una imagen se puede encontrar como: Unavailable (El navegador no obtuvo ninguna informacion), Partially Available (El navegador obtuvo una parte de la imagen), Completely Available (Toda la informacion obtenida correctamente) y Broken (El navegador no puede decodificar la imagen).

    Atributos:

    1. Alt: Establece un texto alternatio que describe a la imagen, mismo que sera utilizado en el caso de que no se pueda renderizar la imagen. Omitir este atributo simboliza que la imagen es parte del contexto del texto, mientras que dejarlo con un valor vacio indica que la imagen no es importante.

    2. Height: Indica en pixeles la altura del imagen.

    3. Width: Indica en pixeles el ancho de la imagen.

    4. Ismap: Valor booleano que indica que la imagen es parte de un mapa del servidor. De ser cierto las coordenadas de un click sobre la imagen se envian al servidor. Solo valido cuando la imagen esta dentro de un vinculo (a).

    5. Longdesc: Especifica un URL que se dirige a un documento externo o seccion interna con la descripcion detallada de la imagen.

    6. Sizes: Lista separada por comas que indica una serie de tamaños para cada dispositivo. (Responsive Design).

    7. Src: Especifica el URL de la imagen y es obligatorio. En el caso de que se utilice Srcset, entonces Src pasa a ser la opcion primaria con una descripcion de densidad de (1x).

    8. Srcset: Utilizada en conjunto con Sizes, es una lista separada por comas de los URL on las imagenes que requiere Sizes para cada dispositivo. Cada imagen incluye una descripcion de densidad para obtener su tamaño relativo.

    9. Usemap: El URL parcial de una imagen mapa asociada con esta imagen. Las imagenes mapa utilizan la etiqueta (map).

      Tags Estructurales

     •  Main •  Section •  Article •  Nav •  Aside •  Footer •  Header •  Address

    Main

    Representa el contenido principal del cuerpo (Body) del documento o aplicacion web. Contiene todo el contenido unico para dicha pagina, excluyendo el contenido que se repide en el sitio, los cuales suelen ser el Header y Footer.

    Section

    Representa una seccion generica, ggeneralmente parte de una serie de secciones a modo de capitulos o partes. Cada seccion, al igual que cada capitulo de un libro, debe de contener un titulo adecuado. Si el contenido es independiente de otras secciones se recomienda utilizar un articulo.

    Article

    Representa una parte individual y completa en la composicion del documento. Cada articulo debe de contar con su propio titulo, tambien es valido tener articulos dentro de otros articulos siempre y cuando su contenido se pueda mantener independiente de otras secciones.

    Nav

    Representa una seccion del documento que contiene los links a otras paginas o partes internas del sitio. Los links de esta seccion no tienen que estar forzosamente en una lista, pueden estar distribuidos en un parrafo por ejemplo.

    Aside

    Representa una seccion cuyo contenido esta relacionado tangentemente con el contenido principal, se utiliza para mostrar notas, anuncios, avisos, links relacionados, perfiles y otra informacion que es importante para el sitio pero que no es fundamental para el contenido principal de la pagina.

    Footer

    Representa el pie de pagina de una seccion o del sitio en general. Usualmente contiene informacion sobre el autor, sobre documentos relacionados, copyright, e informacion legal. Un footer puede ser utilizado como cualquier otra seccion, incluyendo titulos, imagenes, parrafos y demas contenido.

    Header

    Representa el contenido intruductorio a una seccion, documento o sitio. Usualmente contiene un titulo y un parrafo introductorio, tambien suele contener logos y formas de busqueda.

    Address

    Representa la informacion de contacto de una persona con respecto a una seccion, articulo o sitio en general. El elemento no debe utilizarse para otro tipo de informacion que no sea el de contacto con una persona.

      Tipos de Contenido
    En HTML los tipos de contenido se expanden a 7, muchas veces un elemento de informacion puede pertenecer a varios de dichos tipos/modelos de contenido. 1.- Flow Content (Todos exepto tipos especiales de metadata que no se puede mostrar en el buscador) 2.- Phrasing content (Texto, hypervinculo, imagenes, videos, y todas las clases de texto) 3.- Heading content (Todas las clases de encabezado) 4.- Sectioning content (Todas las nuevas clases diferentes de texto y encabezado como articulo, nav etc..) 5.- Embedded content (Todos los hyperviculos y sus clases) 6.- Interactive content (Todo contenido del que se espera que el usuario introdusca informacion) 7.- Metadata content (Todo contenido que relaciona los atributos de otros contenidos, muchas veces de manera externa)

ASCII was the first character encoding standard (also called character set). ASCII defined 127 different alphanumeric characters that could be used on the internet: numbers (0-9), English letters (A-Z), and some special characters like ! $ + - ( ) @ . ANSI (Windows-1252) was the original Windows character set, with support for 256 different character codes. ISO-8859-1 was the default character set for HTML 4. This character set also supported 256 different character codes. Because ANSI and ISO-8859-1 were so limited, the default character encoding was changed to UTF-8 in HTML5. UTF-8 (Unicode) covers almost all of the characters and symbols in the world. All HTML 4 processors also support UTF-8 encoding. The ASCII Character Set ASCII uses the values from 0 to 31 (and 127) for control characters. ASCII uses the values from 32 to 126 for letters, digits, and symbols. ASCII does not use the values from 128 to 255. ________________________________________ The ANSI Character Set (Windows-1252) ANSI is identical to ASCII for the values from 0 to 127. ANSI has a proprietary set of characters for the values from 128 to 159. ANSI is identical to UTF-8 for the values from 160 to 255. ________________________________________ The ISO-8859-1 Character Set 8859-1 is identical to ASCII for the values from 0 to 127. 8859-1 does not use the values from 128 to 159. 8859-1 is identical to UTF-8 for the values from 160 to 255. ________________________________________ The UTF-8 Character Set UTF-8 is identical to ASCII for the values from 0 to 127. UTF-8 does not use the values from 128 to 159. UTF-8 is identical to both ANSI and 8859-1 for the values from 160 to 255.       Seguridad

    El modelo de seguridad en la web se basa en el concepto de origenes, es decir que sistema esta ejecutando, enviando y recibiendo la informacion del servidor. La mayoria de los ataques estan destinados a fingir ser el sistema de origen que deberia ejecutar, enviar o recibir la informacion denominados ataques Cross-Origin.

    No Validar el Input del Usuario, Cross-Site Scripts y SQL Injection son ataques Cross-Origin. Cuando un sitio espera recibir informacion del usuario tiene que tener una lista de tipos de informacion validas y escapar todo input que no se adiera a una estricta serie de reglas. Para mantener estos filtros se recomienda que los objetos que se establescan los atributos validos para cada tipo de objeto en especial si se espera que el usuario introdusca URLs.

    Cross-Site Request Forgery son acciones no autorizadas que un atacante obliga a un cliente a hacer a travez de paginas falsas y otros trucos. Para evitar esto podemos checar el Origin Header de todas las peticiones de informacion. Clickjaquing es tambien comun he involucra la creacion de un iframes en el sitio en la posicion del mouse para redirigir el trafico a un sitio maligno.
